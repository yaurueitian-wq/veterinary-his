## 二、個人資料保護法

> 民國 99 年制定，111 年最新修正

### 2.1 資料分類

| 資料類型 | 個資法分類 | 保護等級 | 說明 |
|---------|---------|---------|------|
| 飼主姓名、電話、地址 | 一般個資 | 中 | 可識別自然人 |
| 飼主身分證號 | 一般個資 | 高 | 唯一識別碼 |
| 飼主付款紀錄 | 一般個資（財務） | 高 | |
| 動物病歷本體 | 非直接個資 | 中 | 與飼主資料連結後整體受保護 |
| 動物影像（X-ray 等） | 非直接個資 | 中 | 同上 |
| 獸醫師操作紀錄 | 非個資（職業行為） | 低 | |

> **關鍵判斷**：動物本身無法律主體地位，動物病歷本身不屬個資法「病歷」定義。但因病歷與飼主資料緊密連結，整個資料庫需依個資法管理。

### 2.2 蒐集告知義務（第 8 條）

蒐集飼主個資時，必須告知：
1. 蒐集機構名稱
2. 蒐集目的（診療、帳務）
3. 個資類別
4. 利用期間、地區、對象及方式
5. 當事人的查詢、更正、刪除等權利

**系統設計要求**：掛號或初診時需記錄告知同意時間與方式（書面 / 口頭 / 電子）。

### 2.3 安全維護義務（第 27 條）

> 非公務機關保有個人資料檔案者，應採行適當之安全措施，防止個人資料被竊取、竄改、毀損、滅失或洩漏。

台灣個資法對「適當安全措施」的定義為**原則性**，無針對獸醫診所的具體技術規範（不同於衛福部對人類醫院的細部規定）。建議對照 ISO 27001 框架實施。

### 2.4 罰則

| 違規事項 | 罰則 |
|---------|------|
| 未盡安全維護義務 | 最高 50 萬新台幣（第 48 條） |
| 故意外洩個資 | 最高 5 年有期徒刑（第 41 條） |
| 集體訴訟損害賠償 | 最高 2 億新台幣（第 29 條） |

## 三、動物診療業管理辦法

> 農業部，約 112 年最新修正（請至農業部全國法規資料庫確認最新版）

### 3.1 病歷必要記載項目

1. 動物種類、品種、性別、年齡
2. 診療日期
3. 主訴（飼主描述症狀）
4. 理學檢查紀錄
5. 診斷或懷疑診斷
6. 治療方式及用藥（含劑量）
7. 診療獸醫師簽名（或識別）

### 3.2 保存年限

| 資料類型 | 最低保存年限 | 備註 |
|---------|------------|------|
| 動物病歷 | **3 年**（需確認是否已修正為 5 年） | 自最後就診日起算 |
| X 光片及醫療影像 | **3 年** | |
| 麻醉及手術紀錄 | **3 年** | |

> 對比：人類醫療病歷依醫療法規定保存 **7 年**（門診）；影像 **15 年**。

### 3.3 電子病歷

動物診療業管理辦法**未規定電子病歷格式標準**，農業部實務上接受電子形式，設計自由度較高。無強制 PKI 電子簽章要求。

### 3.4 獸醫師保密義務（獸醫師法）

獸醫師有保密義務，不得任意洩漏診療過程中得知的飼主資訊，違者可受懲戒。

## 四、管制藥品管理

> 母法：管制藥品管理條例（衛福部食藥署主管）

### 4.1 獸醫診所最常涉及的管制藥品

| 藥品 | 分級 | 常見用途 |
|------|------|---------|
| Ketamine（氯胺酮） | 第三級 | 麻醉誘導、止痛 |
| Xylazine | 非管制（但有管控） | 大型動物鎮靜 |
| Morphine / Fentanyl | 第一 / 二級 | 術後止痛 |
| Pentobarbital | 第二級 | 安樂死 |

### 4.2 法規義務

**申請資格**：獸醫診所使用管制藥品須向衛福部食藥署申請「管制藥品使用執照」，並設置**管制藥品管理人**（通常為負責獸醫師）。

**記錄義務（每次使用必須記錄）**：
1. 使用日期
2. 動物（患者）識別
3. 藥品名稱、批號
4. 使用劑量
5. 剩餘數量（流水帳，每次扣帳）
6. 施用人員簽名

**定期申報**：每兩個月向食藥署申報一次使用量（電子申報）。

**盤點與查核**：帳面數量必須與實物相符，食藥署可不定期稽查。差異須立即通報。

### 4.3 對系統設計的影響

| 需求 | 說明 | 優先級 |
|------|------|--------|
| 管制藥品流水帳 | 每次使用獨立一筆記錄（領用 / 使用 / 餘量），不可修改 | 用藥模組 P0 |
| 批號追蹤 | 記錄每批藥品的批號與有效期 | 用藥模組 P1 |
| 定期申報匯出 | 兩個月彙整報表，格式需符合食藥署規定 | 用藥模組 P1 |
| 盤點功能 | 帳面餘量 vs 實物盤點比對 | 用藥模組 P2 |
| 操作不可修改 | append-only（已有 `is_superseded` 設計精神，此處更嚴格） | 架構設計 |

> **重要**：管制藥品記錄若有塗改或帳面不符，負責獸醫師可面臨執照撤銷或刑事責任，是所有合規項目中**風險最高**的一項。

## 五、動物用藥品與處方箋

> 動物用藥品管理辦法（農業部）、獸醫師法

### 5.1 處方藥 vs 指示藥

| 類型 | 定義 | 獸醫師角色 |
|------|------|----------|
| 處方動物用藥品 | 需獸醫師處方才能使用 | 開立處方箋，記錄於病歷 |
| 指示動物用藥品 | 可依標示使用，不需處方 | 建議使用，但仍記錄於處方 |

### 5.2 處方箋必要記載

1. 動物名稱（或識別號）
2. 飼主姓名
3. 開立日期
4. 藥品名稱（學名或商品名）
5. 劑量、給藥途徑、頻次
6. 療程天數
7. 獸醫師姓名、執照號碼、簽章

### 5.3 對系統設計的影響

- 處方模組需能產出符合格式的處方箋（列印 / PDF）
- 處方箋與就診紀錄關聯，保存年限同病歷（至少 3 年）
- 開立人員需為具執照獸醫師（系統角色驗證）

## 六、食用動物休藥期

> 動物用藥殘留標準（農業部 / 衛福部食品安全衛生管理法）

### 6.1 適用情境

本系統服務對象包含大型動物（牛、馬）。部分動物為食用動物（乳牛、肉牛），用藥後需遵守**休藥期（Withdrawal Period）**，方可屠宰或採乳。

### 6.2 法規要求

- 農業部公告「動物用藥殘留標準」，列出各藥品在各組織（肌肉、脂肪、肝、腎、乳、蛋）的最高殘留限量（MRL）
- 獸醫師開立藥品時，有義務告知飼主休藥期
- 若違反休藥期導致藥物殘留，獸醫師可能面臨連帶責任

### 6.3 對系統設計的影響

| 需求 | 說明 | 優先級 |
|------|------|--------|
| 休藥期提示 | 開立處方時，系統顯示該藥品的食用動物休藥期 | 用藥模組 P1 |
| 休藥期到期日記錄 | 計算並記錄「可屠宰 / 可採乳日期」 | 用藥模組 P1 |
| 告知飼主記錄 | 記錄獸醫師已告知飼主休藥期（合規存證） | 用藥模組 P2 |

> MVP 大型動物場景：若 MVP 不支援用藥模組，此項可暫緩；但一旦用藥模組上線且有食用動物就診，必須一併實作。

## 七、電子發票與稅務合規

> 統一發票使用辦法、財政部電子發票整合服務平台（EINVOICE）

### 7.1 開立發票的義務門檻

| 月銷售額 | 義務 |
|---------|------|
| 超過 20 萬（一般行業） | 需使用統一發票（含電子發票） |
| 連鎖 5 院合計 | 各分院個別計算，或依集團統一設定 |

連鎖動物醫院月營業額通常遠超門檻，**電子發票為必要功能**，不實作結帳模組無法合法開業。

### 7.2 電子發票種類

| 種類 | 適用場景 |
|------|---------|
| B2C 雲端發票 | 一般飼主（個人）付費 |
| B2B 電子發票 | 企業客戶（寵物店、農場等）|
| 紙本發票 | 系統故障時的降級方案 |

### 7.3 串接要求

財政部電子發票整合服務平台（EINVOICE）提供 API：
- 開立發票（B2C 需上傳至平台）
- 作廢 / 折讓
- 中獎通知（B2C 統一發票）

### 7.4 對系統設計的影響

| 需求 | 說明 | 優先級 |
|------|------|--------|
| 發票號碼管理 | 向財政部取得發票字軌，系統管理號碼 | 結帳模組 P0 |
| B2C 雲端上傳 | 每張發票即時上傳財政部平台 | 結帳模組 P0 |
| 折讓 / 作廢 | 退費場景需能作廢並重開 | 結帳模組 P1 |
| 每月對帳 | 系統開立發票與財政部平台核對 | 結帳模組 P1 |
| 降級方案 | 平台斷線時改用紙本，事後補上傳 | 結帳模組 P2 |

> **建議**：結帳模組設計時，先確認診所是否已有電子發票服務廠商（如統一資訊、鑫 e 點等），可能採用 SaaS 套件串接而非自建。

## 八、放射線設備安全

> 游離輻射防護法（核安會）

### 8.1 適用情境

獸醫診所使用 X-ray（X 光機）屬於「醫用游離輻射設備」，須向**核能安全委員會（核安會）**申報。

### 8.2 法規義務

| 義務 | 內容 |
|------|------|
| 設備申報 | X-ray 機設置前申報，取得核可 |
| 操作人員資格 | 操作者需有輻射安全訓練證明 |
| 操作記錄 | 每次使用記錄（日期、操作人員、動物、照射部位） |
| 定期檢測 | 設備每年定期劑量測定 |

### 8.3 對系統設計的影響

| 需求 | 說明 | 優先級 |
|------|------|--------|
| 影像操作記錄 | 每次 X-ray 記錄操作人員、時間、動物 | 影像模組 P1 |
| 操作人員資格標記 | 系統可識別哪些使用者有輻射操作資格 | 影像模組 P2 |

> 此項目對 HIS 系統的直接影響相對較輕，主要是記錄層面；設備申報、人員訓練屬於診所行政作業，不在 HIS 範圍。

## 九、資訊安全標準對照

### 9.1 ISO 27001 關鍵控制項

| 控制域 | 對本系統的要求 | 目前狀態 |
|--------|--------------|---------|
| A.9 存取控制 | 角色型存取（RBAC）、最小權限 | ✅ 已實作（user_roles） |
| A.10 加密 | 傳輸 TLS 1.2+；靜態敏感欄位加密 | 🔶 TLS 已設；靜態加密待評估 |
| A.12 作業安全 | 稽核日誌、備份、漏洞管理 | 🔶 部分實作 |
| A.13 通訊安全 | Nginx TLS；API 認證 | ✅ 已實作 |
| A.14 系統開發安全 | OWASP Top 10 防護、程式碼審查 | 🔶 持續進行 |
| A.16 資安事件管理 | 事件回應程序 | ❌ 尚未建立 |
| A.17 業務持續 | 備份與災難復原計畫 | ❌ 尚未建立 |
| A.18 合規 | 個資法合規 | 🔶 持續進行 |

### 9.2 HIPAA 與台灣個資法比較

| 面向 | HIPAA（美國） | 台灣個資法 |
|-----|-------------|----------|
| 保護對象 | PHI（受保護健康資訊） | 自然人個人資料 |
| 技術保障規範 | 詳細 technical safeguard | 「適當安全措施」（原則性） |
| 資料外洩通報時限 | 60 天 | 無明確法定時限（即時為原則） |
| 強制認證 | BAA 協議 | 無正式認證機制 |
| 違規罰則 | 最高每年 $1.9M USD | 最高 2 億新台幣 |

HIPAA 可作為技術設計的**參考上界**，台灣個資法要求較寬鬆但仍具法律效力。

## 十、資料外洩通報義務

### 5.1 通報對象與時機

| 通報對象 | 時機 | 方式 |
|---------|------|------|
| 受影響飼主 | 發現後即時通知 | 電話 / 書面 / 電子 |
| 主管機關（農業部） | 發現後即時通報 | 書面報告 |

> 注意：台灣個資法目前無 GDPR 72 小時或 HIPAA 60 天的明確法定時限，但「即時通報」已為實務函釋精神。

### 5.2 通報內容

外洩通報應包含：
1. 外洩發生時間（若可知）
2. 外洩資料類型與範圍
3. 受影響人數估計
4. 已採取的緊急措施
5. 後續改善計畫

```sql
        CHECK (incident_type IN ('data_breach', 'unauthorized_access',
                                  'system_intrusion', 'data_loss', 'other')),
    severity                VARCHAR(20)
        CHECK (severity IN ('low', 'medium', 'high', 'critical')),
    description             TEXT NOT NULL,
    affected_owner_ids      INTEGER[],
    discovered_at           TIMESTAMPTZ NOT NULL,
    reported_at             TIMESTAMPTZ,
    notified_owners_at      TIMESTAMPTZ,
    notified_authority_at   TIMESTAMPTZ,
    status                  VARCHAR(20)
        CHECK (status IN ('open', 'investigating', 'contained', 'closed')),
    created_by              INTEGER REFERENCES users(id),
    created_at              TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### 6.4 資料保存期限管理（尚未實作）

- 病歷保存期限：最後就診日 + 3 年（需確認法規最新版）
- 稽核日誌保存：建議 5 年
- 超過期限的資料：軟刪除 → 等待期（30 天） → 硬刪除，並記錄銷毀事件
